# Spoon定期スタジオ iPhoneアプリ化 — Claude Code引き継ぎ書

## プロジェクトの目的
Windows PCで動いている単一HTMLツール「Spoon定期スタジオ」(spoon_teiki_studio.html)を、
iPhoneだけで定期の作成〜Spoonへの貼り付けまで完結できるアプリにする。

**推奨ロードマップ**: フェーズ1=PWA(ホーム画面に追加して使うWebアプリ。Mac・開発者登録不要、既存コード資産をそのまま活かせる)→ フェーズ2=必要ならCapacitor等でネイティブ化を検討。

## 監督体制とバグ責任(最重要)

### 誰がバグを見つけ、誰が直し、誰が監督するか
- **見つける**: ユズ(QA)。ユズには**拒否権**がある——チェックリストを1つでも落ちたら、リリースを止める権限。誰もこれを覆せない(ユーザー本人を除く)
- **直す**: そのコードを書いた本人(レンまたはサクラ)。他人に押し付けない
- **監督**: ミナト。**検収ゲート**を持つ——下の「完成の定義」を全部満たした証拠がなければ、ユーザーに納品させない

### 完成の定義(Definition of Done)— 全部そろうまで「完成」と言ってはいけない
1. 自動テスト全項目グリーン(現在131項目+新規分)
2. **画面の実物確認**: 実際に描画した状態のスクリーンショットで目視検証(ロジックが通っても見た目が壊れている事故を防ぐ)
3. **実機Spoon検証**: コピー→Spoonコメント欄に貼り付け→形が崩れないことを確認(ユズのチェックリスト)
4. 仕様の理解確認: 新機能は実装前にユーザーへ「この理解で合っていますか」を確認済みであること
5. ミナトによる検収記録

### バグゼロ運用
- **ユーザーがバグを見つけた場合、それは個人のミスではなく体制の失敗**として扱う。ミナトが「なぜチェックをすり抜けたか」を分析し、すり抜けた穴をチェックリストに追加する(同じ種類のバグは二度と通さない)
- 修正時は必ず「再現テストを先に書く→直す→テストが通る」の順(直しただけの修正は禁止)
- 「たぶん動く」「軽微なので」でのスキップは一切認めない

### 過去の実例(この体制が必要な理由)
前任開発で実際に起きた事故: テスト全合格の状態で、①アートが実機でななめに崩れる ②幅ロックが出力に効いていない、をユーザーに発見された。原因は「ロジックのテストだけで実物の描画・実機貼り付けを検証していなかった」「仕様の理解を確認せず実装した」こと。上のDoD 2〜4はこの反省から来ている。新チームは同じ轍を踏まないこと。

## 全員が守る「Spoonの絶対仕様」
1. 1行目の行頭スペースは全部無視される → 点字空白(U+2800)アンカーで対処
2. スペースだけの行は消える → ハングルフィラー半角(U+FFA0)で対処
3. 行末側のスペースは消える(文字の左側のみ有効)
4. 使ってよい見えない文字は実績4種のみ: 半角SP / 全角SP / ⠀(U+2800) / ㅤ(U+FFA0)
5. 200文字制限(改行含む)
6. **アートを含む行の詰め物は点字空白で統一**(スペース混在はフォント差でズレる — v3.3で確定した最重要知見)
7. **幅ロックは1行目だけをロック幅まで埋めて末尾に見えないアンカー**(Spoonは最も広い行で全体幅が決まる)
